# 硬编码治理与配置化重构方案

## 1. 背景

当前项目已经具备完整的单关玩法闭环，但部分内容定义分散在生成、输入、UI、物理和关卡系统中。问题并不只是存在数字或字符串，而是同一份业务信息需要在多个位置重复维护。

典型例子是新增一种植物时，除了修改 `PlantKind`，还需要同步修改植物属性、生成逻辑、卡片列表、数字键映射、UI 快捷键显示和具体行为系统。任何一处遗漏都可能造成“卡片存在但无法选择”或“显示参数与实际参数不一致”。

本文档用于指导后续配置化重构，目标是在不改变现有玩法的前提下建立单一数据来源，并为多植物、多僵尸和多关卡扩展做好准备。

## 2. 目标与非目标

### 2.1 目标

- 同一项内容数据只定义一次，UI、输入和玩法系统共同读取。
- 新增植物、僵尸、弹丸或关卡时，尽量不修改无关模块。
- 战斗数值、碰撞尺寸和实际渲染尺寸之间不存在重复且不一致的定义。
- 配置在进入运行时前完成校验，避免无效行号、负数间隔或缺失引用导致游戏中途失败。
- 保持配置强类型，避免使用任意字符串驱动核心玩法逻辑。
- 每个迁移阶段都能独立编译、测试和提交，不进行一次性大改。

### 2.2 非目标

- 不追求消灭所有常量。
- 不把 ECS 调度、状态转换、碰撞组和数学算法改成外部配置。
- 第一阶段不引入完整编辑器、热重载或模组系统。
- 不在配置化过程中顺便改变现有数值平衡和玩法表现。

## 3. 配置边界

判断一个值是否应该配置化时，优先考虑它是否属于“内容”或“规则”。

适合配置化的内容包括：

- 单位名称、价格、生命、速度、伤害、间隔和产出。
- 精灵颜色、尺寸、名牌样式和碰撞尺寸。
- 卡槽顺序、快捷键和默认选中项。
- 关卡初始资源、允许使用的植物和敌人时间线。
- 弹丸速度、寿命、命中策略和物理参数。
- UI 主题、字号、颜色、间距和资源路径。

应继续保留在代码中的规则包括：

- `GameSet` 调度顺序和 Bevy 状态转换。
- Rapier 碰撞组的位分配及系统间消息流。
- `PlantKind`、`ZombieKind`、`ProjectileKind` 等强类型标识。
- 扫掠检测算法中的区间边界和浮点精度判断。
- 测试用输入与断言期望值。

## 4. 当前问题清单

### 4.1 P0：植物定义分散

当前 `src/game/plant.rs` 中的 `PlantKind` 只集中定义了名称、价格、卡片冷却、生命值和颜色。下列参数仍由具体系统单独写死：

- 精灵和碰撞体尺寸。
- 行为周期，例如向日葵产出周期和豌豆射手射击周期。
- 发射点、太阳生成点等局部偏移。
- 向日葵每次产出的太阳数量。
- 植物行为与具体组件之间的映射。

这会导致植物数据和植物行为无法作为一个完整定义查看或校验。

### 4.2 P0：植物卡槽与输入映射重复

`src/game/ui.rs` 和 `src/game/level.rs` 分别维护了卡槽顺序、快捷键和植物类型映射。UI 更新时又存在一份植物到数字的反向映射。

这些映射应收敛为同一个卡片定义，卡片构建、快捷键处理和提示文字都从该定义生成。

### 4.3 P0：僵尸缺少类型定义

`SpawnZombie` 当前只携带行号，所有生成请求最终都会产生同一种普通僵尸。出生偏移、移动速度、生命、攻击间隔、攻击伤害、碰撞体和显示样式都写在 `src/game/zombie.rs` 的生成与行为系统中。

这使关卡时间线无法表达僵尸类型，也无法在不复制生成逻辑的情况下增加新僵尸。

### 4.4 P0：弹丸定义不完整

`ProjectileDefinition` 已经集中了一部分属性，但只包含伤害、寿命和水平速度。以下参数仍散落在弹丸生成与碰撞系统中：

- 精灵颜色和尺寸。
- 碰撞半径、发射初速度和重力缩放。
- 摩擦、弹性、CCD 和穿透策略。
- 世界清理边界。
- 扫掠检测使用的目标半尺寸。

目前逻辑扫掠检测使用的僵尸半尺寸与僵尸实际 Rapier 碰撞体尺寸不同，应改为共享单位碰撞定义，避免视觉命中和物理命中表现分叉。

### 4.5 P0：关卡资源仍是代码内固定数据

`LevelDefinition` 当前只保存僵尸生成时间线，默认实现中直接写入固定的 11 个生成点。初始太阳、默认植物、草坪布局和失败边界分别存在于其他资源或系统中。

一个完整关卡至少应能够描述：

- 关卡标识和显示名称。
- 初始太阳。
- 草坪布局或草坪布局引用。
- 允许使用的植物卡片。
- 默认选中的植物。
- 带僵尸类型、时间和行号的生成时间线。
- 可选的关卡规则覆盖值。

### 4.6 P1：资源路径和表现主题重复

中文字体路径在 UI、植物、僵尸和太阳拾取物中重复加载。UI 和场景名牌的颜色、字号、阴影和偏移也散落在各生成系统中。

应建立统一的资源句柄和主题资源，生成系统只请求语义化样式，例如“植物名牌”或“不可用卡片”，而不是自行组合颜色和字号。

### 4.7 P1：输入说明可能与真实绑定漂移

重开、物理调试、弹丸沙盒和植物选择按键分别在多个系统中处理，操作说明则是一段独立的固定字符串。后续修改按键时，界面提示不会自动更新。

应让输入系统和操作提示读取同一份 `ControlBindings`。调试按键应通过 Cargo feature 或独立调试插件注册，避免混入正式玩法输入。

### 4.8 P1：世界尺寸相关参数没有统一来源

物理墙位置、弹丸清理范围、太阳点击半径、失败判定偏移和僵尸出生偏移都使用独立数值。部分参数可以由 `LawnLayout` 推导，其余应进入相应的玩法配置。

## 5. 目标数据模型

第一阶段建议继续使用 Rust 强类型资源，不急于加载外部文件。先解决单一数据来源，再考虑序列化格式。

### 5.1 植物目录

```rust
pub struct PlantDefinition {
    pub kind: PlantKind,
    pub display_name: &'static str,
    pub price: u32,
    pub card_cooldown: Duration,
    pub health: f32,
    pub visual: UnitVisualDefinition,
    pub collider_half_size: Vec2,
    pub behavior: PlantBehavior,
}

pub enum PlantBehavior {
    SunProducer {
        interval: Duration,
        value: u32,
        spawn_offset: Vec2,
    },
    Shooter {
        interval: Duration,
        projectile: ProjectileKind,
        muzzle_offset: Vec2,
    },
    Blocker,
}
```

行为枚举保持强类型，系统仍负责执行行为；配置只描述行为参数，不使用字符串命令或脚本。

### 5.2 僵尸目录

```rust
pub struct ZombieDefinition {
    pub kind: ZombieKind,
    pub display_name: &'static str,
    pub health: f32,
    pub speed: f32,
    pub attack_damage: f32,
    pub attack_interval: Duration,
    pub engage_range: RangeInclusive<f32>,
    pub spawn_offset_x: f32,
    pub visual: UnitVisualDefinition,
    pub collider_half_size: Vec2,
}

pub struct SpawnZombie {
    pub kind: ZombieKind,
    pub lane: Lane,
}
```

所有僵尸系统通过实体上的运行时组件执行，不应在每帧查询中反复访问字符串键或外部资产数据。

### 5.3 弹丸目录

扩充现有 `ProjectileDefinition`，至少统一以下数据：

```rust
pub struct ProjectileDefinition {
    pub kind: ProjectileKind,
    pub damage: f32,
    pub lifetime: Duration,
    pub visual: ProjectileVisualDefinition,
    pub radius: f32,
    pub motion: ProjectileMotionDefinition,
    pub hit_policy: HitPolicyDefinition,
}
```

`ProjectileMotionDefinition` 可以继续区分路径运动和 Rapier 物理运动。两类弹丸共享生成入口，但由强类型运动定义决定附加哪些组件。

### 5.4 卡片与输入

```rust
pub struct PlantCardDefinition {
    pub slot: u8,
    pub key: KeyCode,
    pub plant: PlantKind,
}
```

卡片列表是唯一顺序来源。UI 根据列表创建卡片，输入系统遍历列表选择植物，快捷键文案由 `slot` 或绑定元数据生成。

### 5.5 关卡定义

```rust
pub struct LevelDefinition {
    pub id: LevelId,
    pub display_name: String,
    pub starting_sun: u32,
    pub lawn: LawnDefinition,
    pub cards: Vec<PlantCardDefinition>,
    pub default_plant: PlantKind,
    pub spawns: Vec<ZombieSpawnDefinition>,
}

pub struct ZombieSpawnDefinition {
    pub at_seconds: f32,
    pub lane: Lane,
    pub kind: ZombieKind,
}
```

载入后必须校验生成时间非负且有序、行号在草坪范围内、默认植物存在于卡片列表中，并且所有种类都能在内容目录中找到。

## 6. 推荐模块结构

在当前项目规模下不宜拆得过细，建议新增以下模块：

```text
src/game/
  catalog.rs       # 植物、僵尸、弹丸的强类型运行时定义
  config.rs        # 通用玩法设置与校验
  assets.rs        # 字体等共享资源句柄
  controls.rs      # 正式玩法和调试输入绑定
  theme.rs         # UI 与场景名牌主题
```

如果后续启用外部配置，再增加：

```text
assets/config/
  gameplay.ron
  plants.ron
  zombies.ron
  projectiles.ron
assets/levels/
  level_01.ron
```

不建议一开始就为每种植物建立单独模块或配置文件。先用目录资源统一数据，内容数量增长后再拆分。

## 7. 运行时配置与外部配置的分层

外部 RON 数据不应直接成为系统查询时使用的最终结构。建议分成两层：

1. `Raw*Definition`：面向序列化，使用秒数、数组、字符串 ID 等易于编写的类型。
2. `*Definition`：经过解析和校验的运行时结构，使用 `Duration`、`PlantKind`、`Handle<Font>` 等强类型。

转换阶段统一完成：

- ID 解析和引用完整性检查。
- 时间、生命、尺寸和速度的范围校验。
- 关卡行号与草坪布局校验。
- 外部资源路径到 Bevy Handle 的解析。
- 错误信息中包含文件、字段和无效值。

这样可以避免每个玩法系统自行处理缺失字段或非法字符串。

## 8. 分阶段实施计划

### 阶段一：建立强类型内容目录

- 新增植物、僵尸和弹丸完整定义。
- 默认数据仍写在 Rust 中。
- 将现有生成和行为系统改为读取定义。
- 保持所有当前数值和行为不变。

验收标准：

- 植物、僵尸和弹丸的内容参数不存在多份来源。
- 扫掠检测与实际单位碰撞尺寸使用同一份定义。
- 现有测试全部通过，并增加目录完整性测试。

### 阶段二：统一卡片和输入映射

- 建立 `PlantCardDefinition` 与 `ControlBindings`。
- UI 卡片、植物选择系统和操作说明读取同一份数据。
- 将沙盒按键迁移到调试插件或 feature。

验收标准：

- 新增卡片只需要修改一份卡片列表。
- UI 显示的快捷键与实际输入始终一致。
- 卡槽重复、按键冲突和默认植物缺失能在初始化时被发现。

### 阶段三：完善关卡定义

- 把初始太阳、可用卡片、默认植物和僵尸类型时间线纳入关卡。
- 将草坪相关边界尽量从布局推导。
- 让重开流程根据当前关卡重新初始化资源。

验收标准：

- `LevelDefinition` 足以完整重建一局游戏。
- 测试可以构造短关卡，不依赖默认 53 秒时间线。
- 非法行号、乱序时间线和未知内容类型会返回明确错误。

### 阶段四：统一资源和主题

- 通过 `GameAssets` 只加载一次共享字体。
- 引入 `UiTheme` 和名牌样式定义。
- 将颜色、字号、间距和层级从具体生成系统迁出。

验收标准：

- 字体路径只有一个来源。
- UI 状态颜色和场景名牌样式可以集中调整。
- 小窗口下的 HUD 不因固定最小宽度而互相覆盖。

### 阶段五：按需求启用外部 RON

- 仅在准备制作多关卡或频繁调数值时实施。
- 增加原始配置结构、加载流程和错误界面。
- 保留一套经过测试的内置默认配置作为开发和测试基线。

验收标准：

- 修改关卡时间线不需要重新编译。
- 缺失或无效配置不会触发难以定位的运行时 panic。
- 配置结构版本变化有明确的迁移策略。

## 9. 测试要求

配置化不能只依赖 `cargo check`。至少应补充以下自动化测试：

- 所有 `PlantKind`、`ZombieKind` 和 `ProjectileKind` 都存在且只存在一份定义。
- 所有卡片引用的植物都存在，槽位和快捷键不重复。
- 所有关卡生成点按时间排序，时间非负，行号有效。
- 默认植物属于当前关卡允许的卡片集合。
- 所有单位的生命、尺寸、间隔和速度处于有效范围。
- 发射器引用的弹丸类型存在。
- 弹丸扫掠半径与生成时使用的半径来自同一字段。
- 重开后资源恢复到当前关卡定义，而不是全局固定默认值。

每个阶段至少运行：

```powershell
cargo fmt --all -- --check
cargo check --locked
cargo test --locked
```

涉及系统参数或插件注册变化时，还需要使用 `cargo run` 做一次启动验证，因为 Bevy 的部分查询冲突只能在运行时初始化系统时发现。

## 10. 风险与控制措施

### 10.1 一次性迁移范围过大

如果同时迁移内容目录、外部文件、UI 和输入，问题很难定位。必须按阶段提交，每个阶段保持玩法可运行。

### 10.2 配置过度抽象

只有一种实现且短期不会变化的算法不应为了配置化引入动态分发。配置描述参数，代码继续表达规则。

### 10.3 外部配置失去类型安全

外部数据必须先解析、校验并转换为运行时强类型资源，玩法系统不能到处使用字符串 ID 和 `Option` 兜底。

### 10.4 表现参数影响碰撞判定

视觉尺寸和碰撞尺寸可以不同，但必须在同一单位定义中明确命名，不能通过某个 Sprite 尺寸隐式推断碰撞范围。

### 10.5 默认值掩盖配置错误

内容引用缺失时应在加载阶段直接失败并报告字段位置，不应悄悄回退为普通僵尸或默认植物。

## 11. 完成定义

当满足以下条件时，可以认为本轮硬编码治理完成：

- 新增一种现有行为类型的植物，不需要修改 UI、输入或关卡系统代码。
- 新增一种僵尸后，关卡时间线可以通过 `ZombieKind` 选择它。
- 植物、僵尸和弹丸的战斗与碰撞参数都有唯一来源。
- 关卡定义能够完整描述初始资源、卡片和生成时间线。
- 操作提示由真实输入绑定生成。
- 字体和主题资源集中加载与管理。
- 所有配置在游戏开始前完成校验。
- ECS 调度和核心算法仍保持清晰、强类型且由 Rust 代码表达。
