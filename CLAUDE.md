# bevy-pvz 项目工作规范

## 提交规范

本项目采用 **Conventional Commits** 风格的 commit message：

```
type(scope): 简短的中文描述
```

### type（必须）

| type | 何时使用 |
|------|----------|
| `feat` | 新功能 / 新特性 |
| `fix` | Bug 修复 / 行为修正 |
| `refactor` | 重构（功能不变） |
| `docs` | 文档变更（README、注释等） |
| `data` | 数据 / 配置内容变更（RON 关卡配置等） |
| `test` | 测试相关 |

### scope（可选，但推荐）

表示影响的主要模块。常用 scope：

- `level` — 关卡逻辑、波次配置
- `zombie` — 僵尸行为、外观、属性
- `plant` — 植物逻辑
- `projectile` — 弹丸、豌豆
- `lawn` — 草坪、棋盘、场景布局
- `physics` — 物理引擎相关
- `ui` — 用户界面
- `render` — 渲染外观
- `input` — 输入、交互
- `pause` — 暂停功能
- `defense` — 防御设施（小推车等）
- `balance` — 数值平衡
- `debug` — 调试工具
- `window` — 窗口设置
- `model` — 模型 / 数据拆分
- `test` — 测试

### description（必需）

- **中文**，祈使语气（"添加"而非"添加了"）
- 简洁明了，控制在 15 字以内
- 小写字母开头（中文则无此要求）

### 示例

```
feat(plant): 添加双头向日葵
fix(zombie): 对齐特殊外观与碰撞箱
refactor(model): 拆分僵尸外观模块
docs: 更新火豌豆与新增僵尸说明
data(level): 合并关卡大波
```

## 工作流程约定

### 写一块，commit 一次

完成一个逻辑上独立的功能块后，立即 commit。不要等到攒了一大堆改动再一起提交。

> "一块代码"的判断标准：
> - 一个功能点完成（如"添加双头向日葵"）
> - 一个 bug 修好
> - 一个重构步骤完成
> - 理想情况：每次 commit 都是能独立编译通过的、语义完整的变更

### 分支策略

- 日常开发直接在 `main` 上小步提交
- 涉及破坏性改动 / 实验性功能时，先建分支
